# LACS Specification Draft

## Status

This document captures the current design decisions for LACS as of April 10, 2026. It is intentionally opinionated and scoped to the current v0 direction. Remaining items are left open at the end for the next review pass.

## 1. Project Summary

LACS, the Linux Agent Control Standard, is a local control plane for Linux systems. It is designed for power users and system administrators who want an agent to manage real OS tasks without granting it arbitrary shell access or root privileges.

The system should feel like a disciplined "AI OS" layer, but it is not a desktop replacement and it is not a general-purpose agent sandbox. It is a typed, audited, rollback-aware execution layer for Linux administration.

The initial platform is Fedora Silverblue because its transactional deployment model aligns with the core safety goals of LACS.

## 2. Product Positioning

LACS is a safe Linux control plane, not a generic chatbot or browser automation layer.

The product wedge is:

- power users first
- system administrators first
- Fedora Silverblue first
- typed actions first
- auditability first
- rollback first

The project should not attempt to win by being broad. It should win by being trustworthy, precise, and hard to misuse.

## 3. Core Architecture

The architecture has three main parts.

### 3.1 `zeroclaw-brain`

The brain is the unprivileged planning layer.

Responsibilities:

- interpret user intent
- request read-only state from the daemon
- generate a plan as typed LACS actions
- explain the plan in user-facing language

Constraints:

- no sudo
- no arbitrary shell execution
- no direct privileged mutation
- no secret access beyond curated credential references

### 3.2 `lacs-shell`

The shell is the user-facing client.

Responsibilities:

- collect user intent
- show plans and previews
- capture approvals
- display progress and results
- render the transaction timeline
- expose rollback helpers

Constraints:

- UI only
- no direct mutation of the system
- no secondary privileged execution path
- no free-form command executor

### 3.3 `lacs-daemon`

The daemon is the trusted privileged executor.

Responsibilities:

- policy enforcement
- risk classification
- authorization checks
- validation and redaction
- preview generation
- execution of typed actions
- transaction persistence
- rollback metadata management

Constraints:

- local-only access
- typed API only
- no arbitrary shell
- one execution path
- bounded output

## 4. Trust Boundaries

The following trust model is fixed:

- untrusted: model outputs, web content, terminal text, user documents, external pages
- trusted: `lacs-daemon`, policy tables, validation, redaction, transaction logging
- semi-trusted: `lacs-shell`, which should not have root

Secrets must never pass through the brain as raw values. They may be referenced only through daemon-managed credential handles.

## 5. Authorization Model

LACS uses the local operating system identity model.

### 5.1 Identity

The caller is the local user running `lacs-shell`. The daemon authenticates the caller via Unix domain socket peer credentials and local OS identity.

### 5.2 Roles

LACS uses local groups to define roles.

- `observer`
  - read-only state, logs, and transaction history
- `dev`
  - observer plus toolbox, Flatpak, and layered-package workflows
- `admin`
  - dev plus services, firewall, hostname, time, locale, users, and groups
- `boot`
  - admin plus deployment lifecycle, pinning, rebase, rollback, kernel arguments, and reboot

### 5.3 Group Mapping

The implementation should map local groups to the role model.

Recommended group names:

- `lacs-observer`
- `lacs-dev`
- `lacs-admin`
- `lacs-boot`

### 5.4 Authorization Rules

- authorization answers whether a user may invoke a profile or action family
- approval answers whether the user consents to the exact request at this moment
- approval does not override authorization
- a parameter change invalidates any prior approval

## 8. Action Contract

Every action must use the same high-level contract.

### 8.1 Shared Envelope

Each action request should carry:

- `action_name`
- `request_id`
- `params`
- `risk_level`
- `requires_approval`
- `preview`
- `execution_state`
- `result`
- `rollback_hint`
- `policy_notes`

### 8.2 Typed Inputs and Outputs

Each action must have:

- a typed request schema
- a typed preview schema
- a typed result schema
- rollback metadata when applicable

### 8.3 No Generic Executor

The system must not expose a generic:

- `RunCommand`
- `ExecuteScript`
- `ApplyTask`
- free-form shell

The safety model depends on the action surface remaining typed and finite.

## 9. Risk Model

Actions are classified into three risk levels.

### 9.1 LOW

Examples:

- `GetSystemState`
- `CollectDiagnostics`
- `ListDeployments`
- `GetServiceLogs`

Policy:

- no approval required
- still recorded in a transaction
- bounded output

### 9.2 MEDIUM

Examples:

- `InstallFlatpak`
- `RemoveFlatpak`
- `RestartService`
- `SetServiceEnabled`
- `ConfigureWifi`
- `SetDnsServers`
- `CreateToolbox`

Policy:

- preview required
- explicit approval required
- exact parameters must match the approval

### 9.3 HIGH

Examples:

- `UpdateSystem`
- `InstallPackages`
- `RemovePackages`
- `RebaseSystem`
- `RollbackDeployment`
- `CleanupDeployments`
- `SetKernelArguments`
- `PinDeployment`

Policy:

- preview required
- explicit approval required
- stronger warning required
- reboot impact must be shown when relevant
- approval must be invalidated if parameters change

## 10. Preview Semantics

The daemon must generate the preview. The shell renders it. The brain may suggest, but it must not authoritatively determine the preview.

### 10.1 Preview Requirements

Every mutating action preview should show:

- action name
- target object(s)
- current state summary
- proposed change
- risk level
- reboot requirement
- rollback availability
- expected side effects
- warnings or irreversible details
- stable request hash

### 10.2 Fedora-Specific Preview Content

Preview content should reflect the actual Fedora Silverblue model where possible:

- deployments
- rollback targets
- pin state
- cleanup impact
- package layering implications
- Flatpak remote and application identity
- toolbox creation or removal details
- service and network effect

### 10.3 Staleness Rules

- if daemon state changes, the preview becomes stale
- if request parameters change, the approval is invalid
- approvals are bound to the request hash

## 11. Execution Model

Mutating actions should run as jobs rather than opaque one-shot calls.

### 11.1 Job Lifecycle

Each mutating action should move through:

- `queued`
- `running`
- `succeeded`
- `failed`
- `canceled`
- `rolled_back`
- `needs_reboot`

### 11.2 Sync vs Async

- read-only actions may return synchronously
- mutating actions should return a job id immediately
- the shell should subscribe to structured progress events

### 11.3 Retry Semantics

- idempotent retries may be deduplicated by the daemon if the request hash matches
- non-idempotent retries must create a new transaction and a new approval
- any parameter change requires re-preview and re-approval

### 11.4 Cancellation Semantics

- queued jobs may be canceled
- running jobs may be cancelable only if the underlying adapter supports safe cancellation
- unsafe cancellation should become a stop-requested state rather than a fake instant kill

### 11.5 Plan Execution

A plan is an ordered list of jobs.

- each action gets its own transaction
- later plan steps do not start until earlier steps succeed
- a failure stops the plan by default
- resuming requires explicit user action or a new plan

## 12. Transaction Model

Every mutating action must create a durable transaction record.

### 12.1 Required Fields

- `tx_id`
- `created_at`
- `actor_type`
- `actor_id`
- `action_name`
- `params`
- `risk_level`
- `approval_id`
- `preview`
- `pre_state_hash`
- `steps`
- `result`
- `post_state_hash`
- `rollback_ref`
- `status`

### 12.2 Recommended Fields

- `duration_ms`
- `attempt_count`
- `policy_reason`
- `warnings`
- `user_visible_summary`

### 12.3 Transaction Lifecycle

The transaction is the source of truth for:

- audit
- recovery
- timeline UI
- debugging
- compliance
- reproducibility

## 13. User Stories

1. A sysadmin checks deployment history, pins a good deployment, and rebases to a new Fedora release.
2. A workstation update is staged, previewed, approved, and rebooted into the new deployment.
3. A bad update is rolled back and the previous deployment remains pinned.
4. A user cleans old deployments after a maintenance window to free disk space.
5. A developer creates a toolbox, enters it, and removes it when done.
6. A maintainer adds a layered package for host integration and reboots into the new deployment.
7. A kernel argument change is staged, validated, and later reset.
8. A service is started, stopped, masked, unmasked, and inspected through an audited flow.
9. A sysadmin opens a firewall service for an internal app, verifies the state, then closes it later.
10. A laptop user changes DNS while traveling and sees the exact network impact before approval.
11. A machine is renamed before being handed to a new user.
12. A lab host is switched to the right timezone and locale before scheduled work begins.
13. NTP is enabled so a fleet stays time-synced.
14. A new local user is created and added to `wheel`.
15. A contractor account is removed and the group membership is cleaned up.
16. A user searches Flatpak apps, checks the remote source, installs one, and later removes it.
17. A maintainer audits Flatpak remotes and sees only approved sources.
18. A support engineer inspects service logs and correlates a crash with a recent layered package change.
19. An admin creates a toolbox for build work and uses it for the life of a task.
20. A reviewer inspects the transaction timeline and sees exactly who approved a rollback, when, and why.

## 16. Error Taxonomy

LACS should classify failures into a small number of operational categories. The daemon should report these categories consistently across all action families.

### 16.1 Validation Failure

Validation failures happen before execution when the request is malformed, incomplete, unsupported, or outside schema constraints.

Examples:

- missing required fields
- invalid package or unit names
- malformed DNS or hostname values
- unsupported parameter combinations
- request hash or approval mismatch

Behavior:

- do not execute
- do not create a running job
- record the failed request attempt
- surface a clear user-facing explanation

### 16.2 Authorization Failure

Authorization failures happen when the local user lacks the role required for the action or profile.

Examples:

- a dev user tries to manage firewall settings
- an observer tries to stage a deployment
- a user outside the required group attempts to invoke a boot-level action

Behavior:

- do not execute
- record the denial
- identify the missing role when it is safe to do so

### 16.3 Policy Denial

Policy denials happen when the request is valid but the daemon policy says the action is not allowed in the current context.

Examples:

- a package is denylisted
- a unit is not on the allowlist
- a remote is not approved
- the action would violate a rate limit

Behavior:

- do not execute
- record the policy reason
- surface the rule class that blocked it

### 16.4 Stale Approval

Stale approval happens when the user approved one exact request, and then the request changed before execution.

Behavior:

- reject the execution
- require a new preview
- require a fresh approval

### 16.5 Execution Failure

Execution failures happen while the daemon is actively running the action and the underlying adapter fails.

Examples:

- rpm-ostree operation fails
- service restart returns an error
- NetworkManager rejects a change
- Flatpak install fails

Behavior:

- mark the transaction failed
- preserve partial state details
- include a bounded error summary
- include rollback hints when available

### 16.6 Transient Infrastructure Failure

These are temporary failures unrelated to the operation itself.

Examples:

- socket interruption
- timeout
- transient systemd or DBus unavailability
- temporary package metadata failure

Behavior:

- mark the transaction failed or deferred, depending on the stage
- clearly indicate retriable vs non-retriable
- if safe, allow a fresh retry with the same approval only for idempotent actions

### 16.7 Cancellation

Cancellation happens when the user stops a queued or running job and cancellation is safe to honor.

Behavior:

- update the job state to canceled
- stop future plan steps
- record the cancel request

### 16.8 Stuck Execution

If the daemon detects repeated action-error cycles or repeated identical observations, it should halt the job rather than loop forever.

This is a safety measure, not a normal user-facing state.

Behavior:

- stop execution
- mark the job as stuck or halted
- report the repeated pattern

### 16.9 Reboot Required

Some actions succeed but require a reboot to take effect.

Behavior:

- mark the transaction `needs_reboot`
- preserve the final status as successful
- surface the reboot requirement clearly in the UI

### 16.10 Rollback Failure

Rollback can fail independently of the original action.

Behavior:

- record the rollback attempt
- preserve the original transaction
- report the rollback-specific failure separately

## 17. Failure Reporting Rules

- the daemon should use typed failure categories rather than free-form strings
- user-facing messages should be concise and actionable
- low-level technical details may be included in an expandable section
- secrets must never appear in failure text
- the failure category should be visible in the timeline and transaction log

## 18. Notes on Practice

Current adjacent tools reinforce this model:

- Cline checkpoints show the value of recoverable, stepwise changes and explicit restore paths.
- OpenHands stuck detection shows the value of halting repeated action-error loops instead of burning cycles forever.
- Browser Use webhook/task status models show the value of structured state and progress reporting.
- Open Interpreter safe mode reinforces that execution should be conservative by default.

## 19. Action Schema Conventions

The action surface is intentionally typed. Each family uses a shared request and result style so the daemon, shell, and transaction log can treat actions consistently.

### 19.1 Common Request Envelope

Every action request should carry:

- `action_name`
- `request_id`
- `params`
- `caller_role`
- `request_hash`

Approval-related metadata is attached separately and must match the exact `request_hash`.

### 19.2 Common Preview Envelope

Every mutating action preview should include:

- `summary`
- `risk_level`
- `current_state`
- `proposed_change`
- `expected_side_effects`
- `reboot_required`
- `rollback_available`
- `warnings`
- `request_hash`

### 19.3 Common Result Envelope

Every action result should include:

- `status`
- `summary`
- `warnings`
- `job_id` for mutating actions
- `needs_reboot`
- `rollback_ref`
- `transaction_id`

### 19.4 Common Result States

The daemon should use these states consistently:

- `queued`
- `running`
- `succeeded`
- `failed`
- `canceled`
- `rolled_back`
- `needs_reboot`

## 30. UI Structure

The shell should be opinionated and sparse. It is not a general-purpose desktop. It is a control surface for plans, previews, approvals, progress, and history.

### 30.1 Layout

The primary UI should have four persistent areas:

- **Intent pane**
  - user enters goals and requests
  - brain responses are shown here

- **Plan pane**
  - the daemon-generated plan is shown step by step
  - each step has risk, preview, and approval status

- **Execution pane**
  - live job progress
  - current state
  - recent output
  - cancellation state

- **Timeline pane**
  - transaction history
  - approvals
  - rollback points
  - job states

### 30.2 Interaction Flow

The UI flow should be:

1. user expresses intent
2. brain proposes a plan
3. daemon generates previews
4. shell shows the previews
5. user approves required steps
6. daemon runs the jobs
7. shell streams progress and final results
8. timeline records the transactions

### 30.3 Primary Screen States

The shell should have these major modes:

- `idle`
- `planning`
- `previewing`
- `awaiting_approval`
- `executing`
- `needs_reboot`
- `failed`
- `rolled_back`

### 30.4 Visual Priorities

The shell should prioritize:

- current risk
- required action from the user
- reboot requirement
- rollback availability
- current execution state

It should not over-prioritize decorative chat chrome.

### 30.5 Preview Rendering

Preview cards should expose:

- the action name
- the target object
- the old value
- the new value
- the risk level
- the side effects
- the approval requirement

High-risk actions should be visually harder to approve accidentally than medium-risk actions.

### 30.6 Progress Rendering

While a job is running, the UI should show:

- current state
- job identifier
- step index when relevant
- bounded progress text
- warnings
- cancellation state

### 30.7 Timeline Rendering

The timeline should present:

- transaction id
- action
- actor
- approval reference
- status
- reboot marker
- rollback marker
- timestamp

### 30.8 Checkpoint Philosophy

The UI should follow the same recoverability principle seen in tools like Cline, but adapted for system administration:

- every meaningful step is recoverable in the transaction history
- the user should be able to inspect, compare, and roll back
- progress should never hide the actual system state

### 30.9 What the UI Is Not

The shell is not:

- a general desktop environment
- a code editor
- a browser automation dashboard
- a terminal replacement
- a hidden background agent
