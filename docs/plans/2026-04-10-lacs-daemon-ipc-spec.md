# LACS Daemon IPC Specification

**Status:** Draft — 2026-04-10

## Scope

This document specifies the three connected pieces of work needed to
close the gap between the current scaffolded state and a functional
end-to-end system:

1. **IPC wire protocol** — framing, message types, connection model
2. **Daemon dispatcher** — connection loop, auth, request handling,
   action execution, job events
3. **Shell IPC client** — replaces `DemoStateClient`; wires
   `plan_intent` and `approve_preview` to the real daemon

Nothing in this spec changes the brain, the existing wire types in
`lacs-types`, or the transaction schema.

---

## 1. IPC Transport

### 1.1 Socket path

Default: `$XDG_RUNTIME_DIR/lacs/daemon.sock`

Fall back to `/run/user/<uid>/lacs/daemon.sock` if `XDG_RUNTIME_DIR`
is not set.

The daemon creates the directory with mode `0700` and binds the socket
with mode `0600` before accepting connections. `bind_unix_listener`
already handles stale socket removal.

### 1.2 Framing

Every message in both directions uses a 4-byte, little-endian `u32`
length prefix followed by a UTF-8 JSON body of that many bytes. There
is no version byte or header — if the framing format ever needs to
change, that is a new protocol version and a new socket path.

```
┌──────────────────────────────────────────────────────────┐
│  length: u32 LE  │  JSON body (length bytes, UTF-8)      │
└──────────────────────────────────────────────────────────┘
```

Maximum message size: 4 MiB. The daemon MUST close the connection if
a message header claims a body larger than 4 MiB.

### 1.3 Connection model

The shell opens one persistent connection per process lifetime. The
connection is kept open and reused for multiple request/response
exchanges. Either side may close at any time; the other side treats an
unexpected close as a `TransientInfrastructureFailure`.

The daemon accepts connections on a `Tokio` async task pool. Each
accepted connection runs in its own task.

## 2. Message Catalog

All messages are JSON objects with a `"type"` string discriminant.

### 2.1 Shell → Daemon

#### `query_state`

Request the system state needed to populate `CuratedState` for the
brain planner.

```json
{
  "type": "query_state",
  "request_id": "<uuid>"
}
```

#### `preview`

Ask the daemon to generate a preview for an action. The daemon
snapshots current state, computes the proposed change, and returns a
`PreviewEnvelope`. It also records a pending transaction in the store.

```json
{
  "type": "preview",
  "request_id": "<uuid>",
  "action_name": "InstallFlatpak",
  "params": { "app_id": "org.gnome.Builder", "remote": "flathub" }
}
```

`request_id` is a UUID generated by the shell. The daemon uses it to
derive the `request_hash` (SHA-256 of `action_name || params_json`,
hex-encoded) and echoes both back in the response.

#### `execute`

Submit an approved request for execution. The `approval_hash` MUST
equal the `request_hash` returned in the matching preview response.

```json
{
  "type": "execute",
  "request_id": "<uuid>",
  "action_name": "InstallFlatpak",
  "params": { "app_id": "org.gnome.Builder", "remote": "flathub" },
  "approval_hash": "<request_hash from preview response>"
}
```

#### `cancel`

Request cancellation of a running job. The daemon transitions the job
to `Canceled` if it has not yet reached a terminal state.

```json
{
  "type": "cancel",
  "job_id": "<uuid>"
}
```

### 2.2 Daemon → Shell

#### `state_response`

```json
{
  "type": "state_response",
  "request_id": "<uuid>",
  "state": {
    "host_name": "silverblue",
    "deployment": "fedora/41",
    "services": ["NetworkManager.service"],
    "flatpaks": ["org.mozilla.firefox"],
    "toolboxes": ["lacs-dev"]
  }
}
```

#### `preview_response`

```json
{
  "type": "preview_response",
  "request_id": "<uuid>",
  "preview": {
    "summary": "InstallFlatpak preview (medium-risk)",
    "risk_level": "medium",
    "current_state": { "installed": [] },
    "proposed_change": { "install": ["org.gnome.Builder"] },
    "expected_side_effects": ["service interruption"],
    "reboot_required": false,
    "rollback_available": true,
    "warnings": ["approval required"],
    "request_hash": "<sha256hex>"
  },
  "transaction_id": "<uuid>"
}
```

#### `job_started`

Sent immediately when the daemon begins executing the approved request.

```json
{
  "type": "job_started",
  "request_id": "<uuid>",
  "job_id": "<uuid>",
  "transaction_id": "<uuid>"
}
```

#### `job_progress`

Sent for each line of output from the executing command. Optional —
the daemon MAY omit these if the action produces no meaningful
incremental output.

```json
{
  "type": "job_progress",
  "job_id": "<uuid>",
  "line": "Installing org.gnome.Builder from flathub..."
}
```

#### `job_completed`

Sent when the job reaches any terminal state.

```json
{
  "type": "job_completed",
  "job_id": "<uuid>",
  "result": {
    "status": "succeeded",
    "summary": "Installed org.gnome.Builder",
    "warnings": [],
    "job_id": "<uuid>",
    "needs_reboot": false,
    "rollback_ref": null,
    "transaction_id": "<uuid>"
  }
}
```

`status` is one of: `succeeded`, `failed`, `canceled`, `rolled_back`,
`needs_reboot`.

#### `error_response`

Returned in place of any expected response when the daemon cannot
fulfill the request.

```json
{
  "type": "error_response",
  "request_id": "<uuid>",
  "category": "stale_approval",
  "message": "approval hash does not match the previewed request"
}
```

`category` values map directly to `FailureCategory` in `lacs-types`:
`validation_failure`, `authorization_failure`, `policy_denied`,
`stale_approval`, `execution_failure`,
`transient_infrastructure_failure`.

## 3. Security Model

### 3.1 Peer credential authentication

On every accepted connection the daemon calls `SO_PEERCRED`
(Linux: `getsockopt(fd, SOL_SOCKET, SO_PEERCRED)`) to obtain the
connecting process's UID without any client-supplied credential.

```rust
// Pseudocode — Tokio UnixStream exposes peer_cred() on Linux.
let ucred = stream.peer_cred()?;  // -> pid, uid, gid
let groups = nix::unistd::getgrouplist(uid_to_username(ucred.uid), ucred.gid)?;
let role = auth::highest_role_from_groups(&groups);
```

The resolved `CallerRole` is attached to every `RequestEnvelope`
processed on that connection. The shell does not supply a role — the
daemon derives it from the OS.

### 3.2 Authorization check

Before generating a preview or executing any action, the daemon checks
that the caller's role is sufficient:

| Action risk level | Minimum role required |
|---|---|
| `Low` | `Observer` |
| `Medium` | `Dev` |
| `High` | `Admin` |

A caller whose role is insufficient receives an `error_response` with
`category: "authorization_failure"`. The connection remains open.

### 3.3 Approval hash binding

The `request_hash` is the SHA-256 of
`action_name + "\x00" + canonical_params_json` (params serialized with
sorted keys), hex-encoded. The daemon computes this on the preview
request and stores it in the transaction record.

On an `execute` request, the daemon calls
`policy::require_fresh_approval(stored_hash, approval_hash)`. A
mismatch returns `category: "stale_approval"` and does not execute.

## 4. Daemon Dispatcher

### 4.1 Bootstrap sequence

```
main()
  DaemonState::bootstrap(config)          // opens SQLite, binds socket
  loop { listener.accept() }
    spawn connection_handler(stream, state)
```

`DaemonState::bootstrap` already exists. The missing piece is
`connection_handler`.

### 4.2 Connection handler

```rust
async fn connection_handler(stream: UnixStream, state: DaemonState) {
    let role = resolve_role(&stream);       // SO_PEERCRED → groups → role
    let mut framed = FramedStream::new(stream);  // 4-byte LE length framing
    loop {
        let msg = framed.recv_message().await?;
        match msg["type"] {
            "query_state"  => handle_query_state(&mut framed, &state).await,
            "preview"      => handle_preview(&mut framed, &state, role, msg).await,
            "execute"      => handle_execute(&mut framed, &state, role, msg).await,
            "cancel"       => handle_cancel(&mut framed, &state, msg).await,
            _              => send_error(&mut framed, "validation_failure", "unknown message type").await,
        }
    }
}
```

`FramedStream` wraps `tokio::net::UnixStream` with `read_exact` for
the 4-byte header and `read_exact` for the body. It lives in
`lacs-daemon/src/transport/framing.rs`.

### 4.3 Preview handler

1. Validate `action_name` is in `KNOWN_ACTIONS` (share the list from
   `lacs-brain` via `lacs-core`). Return `validation_failure` if not.
2. Authorize caller role against action risk level.
3. Build `ActionSpec` by calling the action family factory with
   params (see §5).
4. Compute `request_hash`.
5. Collect current system state snapshot relevant to the action
   (see §6).
6. Compute `proposed_change` from the action spec and params.
7. Call `preview::preview_action(request, current_state, proposed_change)`.
8. Call `TransactionStore::record_previewed(transaction, preview)` to
   persist a pending record.
9. Send `preview_response`.

### 4.4 Execute handler

1. Look up the pending transaction by `request_hash`.
   Return `stale_approval` if not found.
2. Call `policy::require_fresh_approval`. Return `stale_approval` if
   hashes do not match.
3. Authorize caller role.
4. Build `ActionSpec` from params.
5. Generate `job_id` (UUID v4). Send `job_started`.
6. Spawn `execute_action(spec, params, job_id, state)` as a Tokio task.
   Stream `job_progress` lines back on the same connection.
7. On completion send `job_completed` and update the transaction status
   in `TransactionStore`.

### 4.5 State query handler

Runs the state collection commands (see §6) concurrently, assembles
`CuratedState`, and sends `state_response`.

## 5. Action Parameter Model

The action family constructors already accept runtime params:

```rust
flatpak::install_flatpak(app_id: &str, remote: &str) -> ActionSpec
```

The executor resolves params from `RequestEnvelope.params` and calls
the right constructor. This lookup lives in
`lacs-daemon/src/executor.rs`:

```rust
pub fn build_action_spec(
    action_name: &str,
    params: &serde_json::Value,
) -> Result<ActionSpec, ExecutorError> {
    match action_name {
        "InstallFlatpak" => {
            let app_id = require_str(params, "app_id")?;
            let remote = require_str(params, "remote")?;
            Ok(flatpak::install_flatpak(app_id, remote))
        }
        "RemoveFlatpak" => Ok(flatpak::remove_flatpak(require_str(params, "app_id")?)),
        "UpdateSystem"  => Ok(deployment::update_system()),
        "InstallPackages" => {
            let pkgs = require_str_array(params, "packages")?;
            Ok(layering::install_packages(&pkgs))
        }
        // ... all 63 actions
        _ => Err(ExecutorError::UnknownAction(action_name.to_string())),
    }
}
```

Helper functions `require_str` and `require_str_array` extract and
validate fields, returning `ExecutorError::MissingParam` on failure.

Action family constructors that currently use placeholder strings
(e.g., `flatpak::install_flatpak("app-id", "flathub")` in `specs()`)
keep their signatures unchanged — the `specs()` function is only used
for the action catalog (tests, documentation). The executor calls the
constructors directly with real params.

## 6. System State Collection

`lacs-daemon/src/state_collector.rs` — a new module that runs
read-only commands concurrently and assembles `CuratedState`. All
commands are run with `tokio::process::Command` and a 10-second
timeout.

| Field | Command | Parse |
|---|---|---|
| `host_name` | `hostname` | trim stdout |
| `deployment` | `rpm-ostree status --json` | `.deployments[0].version` or `checksum[0..12]` |
| `services` | `systemctl list-units --type=service --state=active --plain --no-legend` | first column of each line |
| `flatpaks` | `flatpak list --app --columns=application` | one app ID per line |
| `toolboxes` | `toolbox list --containers 2>/dev/null` | lines after the header |

If any command fails (binary not found, non-zero exit, timeout), the
field is populated with an empty vec or a fallback string and a
warning is logged. The state query MUST succeed even on non-Silverblue
systems — the brain's safety fence will reject irrelevant actions.

## 7. Action Executor

`lacs-daemon/src/executor.rs` — owns `build_action_spec` (§5) and
`execute_spec`.

```rust
pub struct ExecutorOutcome {
    pub status: JobState,
    pub summary: String,
    pub warnings: Vec<String>,
    pub needs_reboot: bool,
    pub rollback_ref: Option<String>,
}

pub async fn execute_spec(
    spec: &ActionSpec,
    params: &Value,
    progress_tx: mpsc::Sender<String>,
) -> ExecutorOutcome { ... }
```

`execute_spec` pattern-matches on `spec.mechanism`:

- `ActionMechanism::Command { program, args }` — spawns
  `tokio::process::Command`, streams stdout lines to `progress_tx`,
  waits for exit, maps exit code to `JobState`.
- `ActionMechanism::FileScan { path }` — reads the file, sends its
  content as one progress line.
- `ActionMechanism::FileWrite { path, content }` — writes atomically
  (write to `.tmp`, `rename`).
- `ActionMechanism::FilePatch { path, search, replace }` — read,
  replace first occurrence, write back.
- `ActionMechanism::FileDelete { path }` — `fs::remove_file`.

For `Command`, the daemon runs the program directly (no shell
expansion). If `spec.reboot_required` is true and exit code is 0, the
outcome sets `needs_reboot: true` and the job reaches `NeedsReboot`
state.

Rollback hint for `rollback_available` actions: capture the pre-change
state snapshot (e.g., `rpm-ostree status --json` before `upgrade`) and
store it as `rollback_ref` in the transaction.

## 8. Shell IPC Client

### 8.1 `DaemonIpcClient`

New file: `apps/lacs-shell/src-tauri/src/daemon_client.rs`

```rust
pub struct DaemonIpcClient {
    sender: Arc<Mutex<FramedWriter>>,
    pending: Arc<Mutex<HashMap<String, oneshot::Sender<DaemonResponse>>>>,
    events: broadcast::Sender<DaemonEvent>,
}

impl DaemonIpcClient {
    pub async fn connect(socket_path: &Path) -> Result<Self, IpcError>;
    pub async fn query_state(&self) -> Result<CuratedState, IpcError>;
    pub async fn preview(&self, action_name: &str, params: Value)
        -> Result<PreviewEnvelope, IpcError>;
    pub async fn execute(&self, action_name: &str, params: Value, approval_hash: &str)
        -> Result<ResultEnvelope, IpcError>;
    pub fn job_events(&self) -> broadcast::Receiver<DaemonEvent>;
}
```

The client spawns a reader task on connect. Incoming messages are
routed: `*_response` messages resolve pending `oneshot` channels keyed
by `request_id`; `job_*` messages are broadcast on `events`.

### 8.2 `StateClient` implementation

```rust
impl StateClient for DaemonIpcClient {
    fn curated_state(&self) -> Result<CuratedState, PlanningError> {
        // block_on the async query_state, map IpcError → PlanningError::StateUnavailable
        tokio::runtime::Handle::current()
            .block_on(self.query_state())
            .map_err(|e| PlanningError::StateUnavailable(e.to_string()))
    }
}
```

### 8.3 Tauri command changes

`commands.rs` — remove `DemoStateClient` and replace the two
`TODO(daemon-ipc)` stubs:

**`plan_intent`** — unchanged at the brain call site; the planner
now gets a `DaemonIpcClient` as its `StateClient`, so `curated_state()`
goes to the real daemon.

**`approve_preview`** — change from event emit to daemon `execute`
call:

```rust
#[tauri::command]
pub async fn approve_preview(
    state: tauri::State<'_, ShellCommandState>,
    action_name: String,
    params: Value,
    request_hash: String,
) -> Result<(), String> {
    state.daemon
        .execute(&action_name, params, &request_hash)
        .await
        .map_err(|e| e.to_string())?;
    Ok(())
}
```

`ShellCommandState` gains a `daemon: DaemonIpcClient` field.

### 8.4 Job event forwarding

The reader task broadcasts `job_started`, `job_progress`, and
`job_completed` events to a `broadcast::Receiver`. A separate Tauri
task consumes these and emits the corresponding Tauri frontend events:

| Daemon event | Tauri frontend event |
|---|---|
| `job_started` | `lacs:job-started` |
| `job_progress` | `lacs:timeline-entry` |
| `job_completed` | `lacs:job-completed` |

This replaces the current `publish_timeline_event` and
`publish_job_outcome` commands (those were placeholders).

## 9. Transaction Integration

### 9.1 On preview

```rust
transactions.record_previewed(
    NewTransaction {
        request_id,
        request_hash: computed_hash,
        action_name,
        risk_level: spec.risk_level,
        status: JobState::Queued,
        approval_id: None,
        summary: preview.summary.clone(),
        warnings: preview.warnings.clone(),
    },
    preview.clone(),
)?;
```

### 9.2 On execute (job completion)

Update the existing transaction to `Succeeded`, `Failed`, or
`Canceled`. Add `approval_id` (the approval hash) at this point.

There is no separate "update" method on `TransactionStore` today —
add `TransactionStore::update_status(transaction_id, status,
approval_id)` as part of this work.

## 10. Error Handling

| Condition | `FailureCategory` | Connection kept? |
|---|---|---|
| Unknown action name | `ValidationFailure` | yes |
| Missing required param | `ValidationFailure` | yes |
| Insufficient role | `AuthorizationFailure` | yes |
| Approval hash mismatch | `StaleApproval` | yes |
| Command not found | `ExecutionFailure` | yes |
| Command non-zero exit | `ExecutionFailure` | yes |
| Command timeout | `StuckExecution` | yes |
| SQLite error | `TransientInfrastructureFailure` | yes |
| Message too large (> 4 MiB) | — | **closed** |
| Malformed JSON frame | — | **closed** |

Connections are closed only for framing-level failures. All
application-level errors return an `error_response` and keep the
connection open.

## 12. Implementation Order

Work in this order so each step leaves a testable system:

1. `framing.rs` — `FramedStream` with unit tests (encode/decode round-trip)
2. `state_collector.rs` — `collect_curated_state()` with integration test on the
   real system (can be skipped in CI with `#[ignore]`)
3. `executor.rs` — `build_action_spec` with unit tests for every action; `execute_spec`
   integration tests using `echo` / `true` as stand-in commands
4. `transactions.rs` — add `update_status`; update existing tests
5. `dispatcher.rs` — `connection_handler` with integration tests using a
   `UnixStream` pair (`socketpair`)
6. `daemon main.rs` — accept loop; test that the daemon starts and accepts a connection
7. `daemon_client.rs` — `DaemonIpcClient` with integration test against a real daemon
   started as a subprocess
8. `commands.rs` — replace `DemoStateClient`, rewrite `approve_preview`
